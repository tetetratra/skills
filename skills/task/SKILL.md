name: task
description: 与えられたタスクについてロードマップを引きながら自律レベルに従って進める。ユーザーから明示的に使用の指示があった場合にのみ使用すること。
---

## 目的
与えられたタスクを、自らロードマップを引きつつ進めます。
必ず下記の手順通りに作業を進めてください。

## 用語の整理
- ステップ：「要件定義」「実装」などの、タスクを進めるための抽象的な作業単位
- ロードマップ：「要件定義 → 設計 → 実装計画 → 実装」といった、タスクを進めるためのステップの順序付け
- フローチャート：ステップを取捨選択してロードマップを策定するための手順
- スキル：AIエージェントに関する一般の技術用語。なにか作業を行う際の手順や指針が記載されている文章。この文章中においては、1つのステップに対してhas-many(0個も可)でスキルが紐づく

## 手順

1. まず、以下の2点についてユーザーに質問してください
  - 作業における中間成果物（目的・背景・方針案・設計 などの文章）をどこに保存すべきか
    - 選択肢
      a. 保存なし（会話上で完結）
      b. チケット上（GitHub Issues、Notion など）
      c. ローカルファイル（`tmp/{タスク名}.md` などにmarkdownファイルを保存）
  - 今回のタスクではどの自律レベルで動いてほしいか（＝AIにどのくらい自律的に動いてもらいたいか）
    - 選択肢
      - レベル1
        - 意思決定が必要な状況では常にユーザーに確認をとる
        - ステップが完了するごとに、常にユーザーに報告・確認を求めること
      - レベル2
        - ユーザーから承認済みのロードマップがすべて完了し、新しくロードマップを引き直した際にのみユーザーに報告・確認を求める
        - ステップが完了した際のユーザーへの報告・確認はせずに次のステップに進むこと
      - レベル3
        - ステップが完了した際や、ロードマップを引いた際、ユーザーへの報告・確認はせずに次のステップに進むこと
        - ただし、意思決定が必要な状況のうち、AIが自信をもって判断できない状況に限り、次のステップに進む前にユーザーに確認をとる
      - レベル4
        - ステップが完了した際や、ロードマップを引いた際、ユーザーへの報告・確認はせずに次のステップに進むこと
        - 意思決定が必要な状況のうち、AIが自信をもって判断できない状況であっても、すべての意思決定をAIが独断で行う
    - 今後の作業は指示された自律レベルで動いてほしいですが、例外的に、以下の状況に陥った場合に限り作業の手を止めてユーザーからの指示を待って下さい
      - 危険を伴う操作（外部リソースの操作・復元不可能な変更）を実行しようとする場合
      - 作業がロードマップ通りに進まず、手詰まりになった場合
      - 重大なインシデントを起こしたことに気づいた場合

以降、タスクが完了するまで、以下の2〜5を繰り返してください

2. フローチャートに従って、タスクのロードマップを引いてください（途中まででもOK）
  - フローチャートには、「残りロードマップの再検討」というステップを追加して終了するよう指示している箇所があります。
    これは、タスクの具体性が不足している状態で無理に全ロードマップを計画することを避けるためです。
    一般に複雑なタスクでは、前段のステップで得られた情報や結論によって後段のアプローチが大きく変わる可能性があるため、最初の段階で後段のステップをすべて見積もることは困難です。
    そのため、具体性の不足を解消するためのステップの追加後は「残りロードマップの再検討」をしてもらうようにしています。

3. 2で作ったロードマップの各ステップに、そのステップを行う際に利用するスキルを選んでください
  - 今あなた（AI）が見えているスキルから、各ステップに適切なスキルを選んでください
  - 適切なスキルが無いステップについては無理に選ぶ必要はありません
  - この段階で、各ステップについて「ユーザーに表示する際のスキル表記」まで決めてください
    - 例: `背景・目的の整理 (clarify-background スキル)`
    - 適切なスキルが無い場合: `タスク分解 (使用スキルなし)`

4. （自律レベル次第ではスキップ可）引いたロードマップ（スキル付き）をユーザーに提示して、承認を貰ってください
  - その際、フローチャートでどのような分岐を通ってそのロードマップになったのかも添えて提示してください
  - これまでの手順で策定したロードマップは、あくまであなた（AI）目線で引いたロードマップとなります。
    そのため、ユーザー目線からすると「xxxという技術・システムに疎いからまずはそれを知りたい」というような意見が出る場合があります。
    その場合、たとえば「（ユーザーの）技術理解の促進」のようなステップ（や、スキル）をロードマップに追加するよう提案してください。
  - また、早とちりして次のステップの具体的な内容についての相談とかはレスポンスに含めないでください（実際の作業は手順の 5 に移ってから初めて実施してほしいです）
  - ロードマップをユーザーに見せる際は、**各ステップの行に必ず使用スキルを併記**してください
    - スキル表記はステップ名と同じ行に書いてください
    - スキル名を別セクションや脚注に追い出してはいけません
  - ロードマップに併記する「状態」は、基本的には `完了`, (ユーザーからの)`確認待ち`, `未着手` から選んでください
  - 提示例1
    ```
    ## ロードマップ提案
    - 背景・目的の整理 (clarify-background スキル)
      - 理由：タスクで達成したいことが未整理なため
      - 状態: 未着手
    - 残りロードマップの再検討 (task スキル)
      - 状態: 未着手
    ```
  - 提示例2
    ```
    ## ロードマップ提案
    - 既存システム・関連技術の調査 (explore-solutions, explain-technology スキル)
      - 理由：タスクで達成したいことが未整理なため
      - 状態: 完了
    - 設計 (design-architecture スキル)
      - 目的: 実装方針と責務分割を決める
      - 状態: 確認待ち
    - 実装計画 (plan-implementation スキル)
      - 目的: 合意済み設計に従って実装計画を立てる
      - 状態: 未着手
    - 実装 (implement スキル)
      - 目的: 実装計画に従ってコードを変更する
      - 状態: 未着手
    ```
  - 以下のような、スキル表記の無いロードマップはNGです
    ```
    ## ロードマップ
    - 設計
      - ...
    ```

5. ロードマップに従って作業を進めてください
  - 作業を進めて「残りロードマップの再検討」に到達した場合、2の手順に戻ってください
  - ロードマップを再掲・更新する場合も、各ステップのスキル表記を必ず維持してください
  - 全ての作業が完了し、最初に与えられたタスク全体自体が完了した場合、ゴールです。ユーザーに結果を報告してください

## ロードマップ策定のためのフローチャート

フローチャートの実行中、以下の行為は避けてください。
- 時間のかかる調べ物
- ユーザーへの質問

1. 与えられた依頼やチケット中に、タスクの背景や目的が書いてありますか？以下のチェックリストで確認してください
  - チェックリスト
    - タスクのビジネス的な意義・必要性が書いてある
    - タスクが起票された経緯が書いてある
    - タスクで達成したいことと、その理由が書いてある
  - すべてチェックできる: 次へ
  - 1つでもチェックできない: ロードマップに「背景・目的の整理」「残りロードマップの再検討」を追加し、フローチャートを終了する

2. 今回のタスクを進めるために、あなた（AI）は以下の情報を持っていますか？もしくは既にどこかに情報がまとまっていますか？以下のチェックリストで確認してください
  - チェックリスト
    - 関連するシステムの動作フローを理解している or まとめ文書がある
    - データ構造やテーブル構成を把握している or まとめ文書がある
    - 使用する技術スタックの特性を理解している or まとめ文書がある
    - 類似の実装例や前例を知っている or まとめ文書がある
  - すべてチェックできる: 次へ
  - 1つでもチェックできない: ロードマップに「既存システム・関連技術の調査」「残りロードマップの再検討」を追加し、フローチャートを終了する

3. 「要求定義」「要件定義」「対応方針決め」のうち、未実施かつ非自明なものはありますか？
  - 判断基準
    - 要求定義：タスクの要求事項(ビジネス的に求められている事柄)がすでに定義されていますか？もしくは自明(= タスクの目的そのまま)ですか？
    - 要件定義：タスクの要件(どのような仕様にするか)はすでに定義されている、もしくは自明(= タスクの目的そのまま)ですか？
    - 対応方針決め：タスクを実現するための大まかな方針は既に決まっている、もしくは自明（＝選択肢が明らかに1つしか無い）ですか？
  - ない（＝すべて実施済みもしくは自明）：次へ
  - ある（＝未実施かつ非自明なものがある）：それらをその順序でロードマップに追加し、末尾に「残りロードマップの再検討」を追加し、フローチャートを終了する

4. タスクを完遂させるまでに残り必要なステップはありますか？
  - ステップ例
    - 汎用
      - 問いの明確化: 漠然とした問いやイシューを、仮説・検証のサイクルを通じて具体的な結論を導く形に整える
      - 背景・目的の整理: なぜこのタスクが必要なのか、何を達成したいのかを明確にする
      - 既存システム・関連技術の調査: 今回のタスクに関連する「システムの仕組み」「技術・概念」「前例」などについて調査をし、意思決定の判断材料を揃える
      - 既存システムの理解促進: 今回のタスクに関連するシステムの仕組みについて、ユーザーの理解を深めさせる
      - 関連技術の理解促進: 今回のタスクに関連する技術や概念について、ユーザーの理解を深めさせる
      - 要求定義: 要求を確定する。ユーザーに提案し、合意を得る
      - 要件定義: 機能要件・非機能要件を定める。要件をユーザーに提案し、合意を得る
      - 対応方針決め: 解決策を検討し、ユーザーに提案し、対応方針を決定する
    - 開発系
      - 技術選定: どの技術・ライブラリ・アーキテクチャを採用するかを決定する
      - 設計: システム設計、データモデル設計、API設計などを行う
      - 実装計画: どのコード（テストコード含む）をどの順番で、どれくらいの段階を踏んで実装していくかを計画する
      - 実装: 実際にコードを書く
      - 動作確認: 検証環境やステージング環境で実際の動作を確認する
      - PR作成: 実装内容をPRにまとめる
      - レビュー修正: レビューを受けてその対応を行う
      - ドキュメント作成: README、設計書、運用手順書などを作成する
    - その他
      - タスク分解：複数の大きなタスクに分解できるような場合に、適切な粒度で子タスクとして分解する
      - 残りロードマップの再検討：これまでに整理した情報をもとに、残りのロードマップを策定できるところまで策定する
  - ある：それらのステップをロードマップに追加してください
  - ない：フローチャートを終了してください

## チェックポイント
- ロードマップを表示するすべての箇所で、各ステップ行に使用スキルが併記されていること
- スキル表記を別セクションに逃がさず、ステップ名と同じ行に書いていること
- 適切なスキルが無い場合でも `使用スキルなし` と明記していること
