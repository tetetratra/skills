タスクに対する解決方針を体系的に検討し、実装アプローチを決定します。

**注意**
このコマンドは技術的な詳細設計を行うものではありません。
比較・検討すべき観点を整理し、方針を決めることが目的です。
詳細な技術設計は以降のステップで行います。

## 目的

- タスクの解決アプローチを観点・軸・比較対象で構造化し、推奨方針を決める
- 高レベルな対応方針を共有し、詳細設計への接続を明確にする

## 手順

### 1. 思考

1. **観点の洗い出し**: 対象の事象について、比較・検討すべき観点を挙げる
   - 例：コスト、開発工数、運用負荷、リスク、将来の拡張性 など
   - 観点の数や内容は事象に応じて変わる。重要度の高い観点を漏れなく拾う
2. **観点ごとの軸の整理**: 各観点について、比較に使う軸（評価項目）を決める
   - 例：「コスト」観点 → 月額、初期費用、内訳、情報ソース
   - 例：「開発工数」観点 → 必要な人日、必要スキル、既存資産の流用可否
3. **比較対象の列挙**: 各観点について、比較対象を挙げ、それぞれの軸に対する評価を整理する
   - 比較対象が1つしかない観点はそれでもよい
   - 検討したが採用しなかった選択肢があれば、不採用理由とともに記録する
4. **推奨案の決定**: 上記の観点・軸での比較結果を踏まえて推奨案を明示し、推奨理由を簡潔に述べる（条件付き可）
   - 推奨案であってもデメリットがあれば記載する

思考フェーズで整理した内容を、以下の整形ステップで出力形式にまとめる。

### 2. 整形

1. **書式の選択**: 観点ごとに、箇条書きと表のどちらが適切かを判断する
   - **箇条書き**: 定性的で、文章で伝えた方がわかりやすいもの（方針の背景説明、深い考察が必要なもの）
   - **表形式**: 定量的に比較できるもの、浅い比較で済むもの、比較対象が複数あるもの。一番右の列に「備考」列を設ける（空でもよい）。備考には補足的な事実情報を書き、良し悪しの判断は含めない
2. **出力**: `## 対応方針` 配下に、観点ごとにセクションを分けて配置する
3. **自己チェック**: 下記チェックリストで冗長さと妥当性、リンクの有無、粒度を確認する

ユーザーからフィードバックがあり、新しい方針案を設ける場合は、(既出の方針案を上書きするのではなく) 新たな案として追加してください。

## 成果物例

以下のフォーマットで出力してください（内容はあくまで例示です）

- 観点A,Bは定性的で方針の背景や考察を文章で伝える必要があるため箇条書きで書いています。箇条書きの場合、軸（メリット・デメリット等）は比較対象ごとのネスト内に記述します
- 観点Cは比較軸ごとに短く比較できるため表形式で書いています。表の場合、列=軸、行=比較対象です
- 推奨の示し方は、箇条書き・表いずれの場合も、観点の末尾に `- 推奨：xxx` の箇条書きで示しています

```
## 対応方針

### 観点A：認証方式について
- 案1：[OAuth 2.0](https://oauth.net/2/) + JWT
  - 方針：標準プロトコルを採用し、トークンベース認証で実装。既存セッション認証と並行稼働させる
  - メリット
    - トークンベースのためネイティブアプリからも同じ認証フローを使え、モバイル対応が容易
    - セッションをサーバー側で持たないため、サーバー台数を増やしてもセッション共有の考慮が不要
    - OAuth 2.0は広く普及しており、ライブラリやドキュメントが豊富で実装コストが低い
  - デメリット
    - トークンの有効期限管理やリフレッシュトークンの実装が必要で、セッション方式より実装が複雑

- 案2：SAML
  - 方針：エンタープライズ向け基盤を導入し、SSOで統合
  - メリット
    - エンタープライズ環境では事実上の標準であり、既存の社内IdPとの互換性が高い
  - デメリット
    - XMLベースでペイロードが大きく、モバイルアプリでの実装が複雑になりやすい
    - リダイレクトフローに依存するため、SPAやネイティブアプリとの統合に技術制約が多い

- 推奨：OAuth 2.0 + JWT。将来性とモバイル対応を優先。標準プロトコルでレビュー工数も抑えられる

### 観点B：データベース設計について
- 案：既存Userテーブル拡張 + 認証情報を別テーブル化
  - 方針：`users`テーブルに最小限のカラム追加、認証トークンは`oauth_tokens`テーブルで管理

- 推奨：既存Userテーブル拡張 + 認証情報を別テーブル化。既存データとの整合性を保ちつつ、認証情報を分離でき、マイグレーションリスクが低い

- 検討したが採用しなかった選択肢
  - 完全新規テーブル：既存データ移行が大規模になり、リスクが高い
  - NoSQL移行：既存RDBMSとの整合性が困難、学習コストが高い

### 観点C：ロールアウト戦略について

| 項目 | 段階的ロールアウト | Blue-Green Deployment | 備考 |
|---|---|---|---|
| 方針 | Feature flagで特定ユーザー→部署→全体の順に展開 | 本番環境を複製し、DNS切り替えで全面移行 | |
| 追加の実装コスト | 小（Flipper導入済み） | 大（環境複製・同期の仕組みが必要） | |
| ロールバック | フラグOFFで即座に可能 | DNS切り戻しで可能だがデータ同期に注意 | |
| 新旧並行稼働 | あり（期間が長くなりやすい） | なし（一括切り替え） | 並行稼働中は両コードパスの保守が必要 |
| インフラコスト | 追加なし | 約2倍（2面維持） | |

- 推奨：段階的ロールアウト。Flipper導入済みで低コスト・低リスク

- 検討したが採用しなかった選択肢
  - カナリアリリース：Kubernetesインフラが未整備のため採用困難
```
