## 目的

漠然とした問いに対して、仮説検証のサイクルを回すことで明確な結論を導く。

問いが漠然としている場合はまず問い自体を明確化し、明確化された問いに対しては具体的な打ち手を検討する、という段階を踏む。

## 進め方

- 各ステップは対象によって具体的な進め方が異なる
- 作業に着手する前に、フローチャートのトレース（現在地と次にやること）とともに作業内容を提示し、ユーザーの承認を得てから実行すること

## 手順

### フェーズ1: 仮説立案

- 1. イシューの特定
  - イシューを立てる（粗くてもよい）

- 2. 仮説の立案
  - if イシューに対して仮説を立てられるか？
    - 仮説を立てる
    - フェーズ2へ
  - else
    - 前提知識・事例を調査する（`explain-technology` や `research-case-study` の活用を推奨）
    - if イシューの解像度が上がった？
      - イシューを更新する
      - フェーズ1からやり直す
    - else if 課題パターンが見えてきた？
      - それを仮説とする（＝見えてきた課題パターンが自身に当てはまっているか？という仮説を立てる）
      - フェーズ2へ
    - else
      - イシューの解像度が上がらない理由を分析する（フローチャートは止めて、ユーザーに相談する）

### フェーズ2: 検証

- 3. 仮説に対して「これが正しいなら、何が観測できるはずか？」という確認項目を出す
  - 例：
    - イシュー「社内のオブザーバビリティは現状十分か？」
    - 仮説「a, c, f の課題が存在する」
    - 観測できること：
      - aの課題があるならば、ログにxというパターンが出ているはず
      - cの課題があるならば、メトリクスにyという傾向が出ているはず

- 4. 確認項目について、実際にそれを確認する作業を行い、仮説が正しいかどうかを検証する

- 5. 仮説が正しいかどうかを踏まえて、立てたイシューに対して結論を出す
  - 例：
    - イシュー「社内のオブザーバビリティは現状十分か？」
    - 仮説「a, c の課題が存在する」
    - 結論「a の課題が存在する」
