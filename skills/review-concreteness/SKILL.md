## 目的

与えられた資料に対して、「地に足ついていない」箇所（= 抽象的・実行不能・根拠不足な記述）を機械的に検出し、具体的な改善案を提示します。

## 手順

### Step 1. 資料のメタ情報を抽出する

資料を全文読み、以下の4つを資料内から抜き出す。資料内に明記されていない場合は「不明」とする。

- **種類**: この資料の種類は何か（調査、提案、企画、報告、設計 等）
- **目的**: この資料は何のために書かれたか
- **想定読者**: 誰に向けて書かれているか
- **主張（結論）**: この資料が最終的に言いたいことは何か

→ 「目的」「想定読者」が不明の場合、それ自体を指摘事項として記録する。
→ 「主張（結論）」については、資料の種類によって扱いが異なる:
  - **調査・比較資料の場合**: 結論がなくてもOK。調査資料の目的は判断材料の提供であり、結論を出すことではない
  - **提案・企画・報告資料の場合**: 結論がなければ指摘事項として記録する

### Step 2. 資料内のすべての「主張文」を抽出する

資料を1文ずつ走査し、以下のいずれかに該当する文を「主張文」として抽出する。

- 「〜すべき」「〜が必要」「〜が重要」「〜を目指す」のような価値判断を含む文
- 「〜する」「〜を実施する」「〜を推進する」のようなアクション宣言を含む文
- 「〜が課題である」「〜が問題である」のような現状認識を述べる文

### Step 3. 各主張文に対して、以下の判定を順に行う

抽出した主張文1つ1つに対して、判定A〜Gを順番に適用する。ただし判定Gは主張文に限らず、資料内の数値・ファクト記述全般に対して適用する。

#### 判定A: 曖昧ワード判定

以下のワードリストに該当する語が、**具体的な定義や数値を伴わずに**使われている場合、指摘する。

> 推進 / 強化 / 向上 / 最適化 / 連携 / 活用 / 促進 / 効率化 / 改善 / 充実 / 高度化 / 加速 / 展開 / 整備 / 実現 / 確立 / 構築（対象が不明な場合）

**判定基準**: 同一文または直後の文に、「具体的に何をどうする」の記述があるか？
- ない → 指摘（カテゴリ: 具体性の欠如）
- ある → OK

良い例:

```
- 指摘前: 「社内コミュニケーションを強化する」
- 改善案: 「週次の部門横断定例を新設し、各チームの進捗と課題を共有する場を設ける」
```

#### 判定B: 5W1H欠落判定

アクション宣言文（「〜する」「〜を実施する」等）に対して、以下の4項目が**同一セクション内に**記載されているかを確認する。

| 項目 | 確認する内容 |
|---|---|
| Who | 誰が（担当者・チーム・部署） |
| What | 具体的に何を（対象と成果物） |
| When | いつまでに（期限・マイルストーン） |
| How | どうやって（手段・手順） |

**判定基準**: 4項目のうち2つ以上が欠けている場合 → 指摘（カテゴリ: 実現可能性の欠如）

良い例:

```
- 指摘前: 「新規顧客獲得施策を実施する」
- 改善案: 「マーケティングチームが、2026年Q3までに、既存顧客のリファラルプログラムを設計・リリースし、月間リード数を{現状値}から{目標値}に引き上げる」
```

#### 判定C: So What? 判定

現状認識文（「〜が課題である」等）や事実の記述に対して、以下を確認する。

**判定基準**: その文の後に、「だからどうするのか」（対応策・結論・示唆）が同一セクション内にあるか？
- ない → 指摘（カテゴリ: 論理構造の不備）
- ある → OK

良い例:

```
- 指摘前: 「直近1年で問い合わせ件数が30%増加している」（記述がここで終わっている）
- 改善案: 「直近1年で問い合わせ件数が30%増加している。現在のCS3名体制では1人あたりの対応件数が限界を超えており、1名増員もしくはFAQの自動応答導入が必要である」
```

#### 判定D: Why So? 判定

価値判断文（「〜すべき」「〜が重要」等）に対して、以下を確認する。

**判定基準**: その文の前後に、そう言える根拠（データ、事実、事例、論理的説明）が1つ以上あるか？
- ない → 指摘（カテゴリ: 根拠・エビデンス不足）
- ある → OK

良い例:

```
- 指摘前: 「モバイルアプリの刷新が急務である」
- 改善案: 「モバイルアプリの刷新が急務である。理由は以下の通り: (1) アプリストアの評価が直近半年で4.2→3.1に低下、(2) DAUが前年比25%減少、(3) 競合X社が2025年にアプリ刷新後、DL数を2倍に伸ばしている」
```

#### 判定E: バズワード判定

以下のようなカタカナ語・流行語が使われている場合、**その語が同一セクション内で具体的に定義されているか**を確認する。

> シナジー / イノベーション / DX / トランスフォーメーション / アジャイル / エコシステム / ソリューション / パラダイムシフト / サステナブル / レジリエンス 等

**判定基準**: その語が指す具体的な内容（何を・どうやって）が同一セクション内に書かれているか？
- ない → 指摘（カテゴリ: バズワード・空疎な表現）
- ある → OK

良い例:

```
- 指摘前: 「DXを推進し、業務効率を高める」
- 改善案: 「受発注業務で現在使用しているFAX・電話ベースのフローを、Webフォーム+自動メール通知に置き換え、受注処理の所要時間を{現状値}から{目標値}に短縮する」
```

#### 判定F: 読み手適合判定

Step 1で特定した想定読者に対して、各主張文の粒度が適切かを確認する。

**判定基準**:
- 想定読者が経営層 → 現場の技術詳細に紙面を割いているのに、ROI・戦略的意義・ビジネスインパクトが書かれていない場合 → 指摘
- 想定読者が現場担当者 → 大局的なビジョンだけで、具体的な手順・ツール・実行計画が書かれていない場合 → 指摘
- 想定読者が不明 → Step 1で既に指摘済みなのでスキップ

良い例:

```
（想定読者が経営層の場合）
- 指摘前: 「Kubernetes上にサービスメッシュを導入し、Istioでトラフィック制御を行う」
- 改善案: 「インフラ基盤を刷新し、サービス間通信の障害検知と自動復旧を可能にする。これにより年間のシステム障害によるダウンタイムを{現状値}時間から{目標値}時間に削減し、機会損失{金額}円の回避を見込む」
```

#### 判定G: 情報源判定

資料内で具体的な数値・ファクト（料金、性能値、統計データ、外部サービスの仕様等）が記述されている箇所に対して、その**情報源（URL、文献名、公式ドキュメントへの参照等）が付記されているか**を確認する。

**判定対象**: 以下のいずれかに該当する記述
- 外部サービスの料金・プラン情報（例: 「Muxは月額$100〜」）
- 性能・スペックに関する数値（例: 「エンコード時間が5-10倍」）
- 市場データ・統計情報（例: 「MAUで1万程度」は自社情報なので対象外）
- 外部サービスの仕様・制約（例: 「FairPlay未対応」）

**判定基準**: 情報源（URL、文献名、「〇〇公式ドキュメントより」等の参照）が同一セクション内またはセクション末尾の参考情報に記載されているか？
- ない → 指摘（カテゴリ: 情報源の欠如）
- ある → OK

**注意**: 自社の内部情報（自社のMAU、自社チームの人数等）や、一般的に自明な事実（「S3はAWSのストレージサービス」等）は判定対象外とする。

良い例:

```
- 指摘前: 「CloudFront Businessプラン: $200/月（50TB転送含む）」（情報源なし）
- 改善案: 「CloudFront Businessプラン: $200/月（50TB転送含む）（参照: https://aws.amazon.com/cloudfront/pricing/ 、2026年4月時点）」
```

### Step 4. 改善の規模を判定する

Step 3で記録した指摘事項の量・種類から、改善の規模を判定する。

| 規模 | 判定基準 |
|---|---|
| **軽微** | 指摘が少数（目安: 3件以下）で、すべて同一カテゴリ内に収まる |
| **構造的** | 以下のいずれかに該当する場合: (1) 複数カテゴリにまたがる指摘がある、(2) 資料のメタ情報（目的・想定読者）自体が不明、(3) 指摘が資料の大部分に及ぶ |

### Step 5. 規模に応じた改善案を提示する

#### 軽微の場合

指摘事項ごとに、箇所単位の書き換え案を提示する。

- 改善案は、欠けている要素を補った書き換え例文を提示する
- レビュアー側が知り得ない情報（具体的な数値等）が必要な場合 → `{ここに〇〇の実測値を入れる}` のようにプレースホルダーで示す

#### 構造的の場合

箇所単位の修正では資料全体の質が上がらないため、**資料をゼロから書き直す場合の推奨アウトライン**を提示する。

推奨アウトラインには以下を含める:
- 資料全体の推奨構成（セクション構成と順序）
- 各セクションで「何を・どのレベルで書くべきか」のガイド
- Step 3の指摘事項が解消されるように設計すること

指摘事項の一覧も併せて出力する（読み手がアウトラインの根拠を理解できるように）。ただし指摘事項に箇所ごとの改善案は付けない（アウトラインが改善案の役割を果たすため）。

## 出力例

### 出力例1: 軽微なケース

架空の「社内Wiki移行のお知らせ」を分析した場合の出力例です。

```
## 分析結果: 社内Wiki移行のお知らせ

### メタ情報
- **種類**: 報告
- **目的**: 社内Wikiの移行スケジュールと対応事項を全社に周知する
- **想定読者**: 全社員
- **主張（結論）**: 2026年6月末までにConfluenceからNotionへ移行する
- **改善規模**: 軽微

### 指摘事項

#### 1. [具体性の欠如] 「旧Wikiのデータを移行する」の範囲が不明確
- **該当箇所**: 「旧Wikiのデータを新環境に移行します」
- **問題**: 「データ」が全ページなのか、特定のスペースのみなのか、添付ファイルを含むのかが不明
- **改善案**: 「旧Confluenceの全スペース（{X}スペース、約{X}ページ）を新環境（Notion）に移行します。添付ファイル（PDF・画像）も含みます。ただし、最終更新が2024年1月以前のページはアーカイブ扱いとし、移行対象外とします」

#### 2. [情報源の欠如] 移行ツールの制約に情報源がない
- **該当箇所**: 「移行ツールの制約により、表組みの一部が崩れる可能性があります」
- **問題**: どの移行ツールを使うのか、どのような制約があるのかの情報源が示されていない
- **改善案**: 「移行ツール（Notion公式インポーター）の制約により、Confluenceマクロ（expand、status等）は変換されず、表組みのセル結合も解除されます（参照: https://www.notion.so/help/import-data-into-notion 、2026年4月時点）」
```

### 出力例2: 構造的なケース

架空の「営業DX推進計画書」を分析した場合の出力例です。

```
## 分析結果: 営業DX推進計画書

### メタ情報
- **種類**: 提案
- **目的**: 営業部門のDX推進について経営会議で承認を得る
- **想定読者**: 経営層
- **主張（結論）**: 不明 — 資料内に明記されていない。「営業DXを推進すべき」という方向性は読み取れるが、具体的に何の承認を求めているのかが不明確
- **改善規模**: 構造的

### 指摘事項

#### 1. [具体性の欠如] 「営業プロセスの最適化」が具体化されていない
- **該当箇所**: 「営業プロセスを最適化し、生産性を向上させる」
- **問題**: 「最適化」「向上」が具体的な定義なく使われている。何をどう変えるのかが読み取れない

#### 2. [実現可能性の欠如] SFA導入のアクションに実行計画がない
- **該当箇所**: 「SFAツールを導入し、顧客管理を一元化する」
- **問題**: Who（誰が推進するか）、When（いつまでに）、How（導入ステップ）が記載されていない。4項目中3項目が欠落

#### 3. [根拠・エビデンス不足] 「DX推進が急務」の根拠がない
- **該当箇所**: 「競争環境の変化に対応するため、営業DXの推進が急務である」
- **問題**: 「急務である」という価値判断に対して、それを裏付けるデータや事実が示されていない

#### 4. [バズワード・空疎な表現] 「DX」が定義されないまま多用されている
- **該当箇所**: 資料全体で「DX」が12回使用されているが、具体的に何を指すか定義されていない
- **問題**: 読み手によって「DX」の解釈が異なるため、この資料で何を実現しようとしているのか合意が取れない

#### 5. [論理構造の不備] 課題の提示が「だからどうする」に繋がっていない
- **該当箇所**: 「営業担当者の残業時間が月平均30時間を超えている」
- **問題**: 事実の提示で終わっており、この課題に対する対応策や示唆が続いていない

#### 6. [読み手適合] 経営層向けなのにビジネスインパクトが不足
- **該当箇所**: 「APIでシステム間連携し、データパイプラインを構築する」
- **問題**: 想定読者は経営層だが、技術的な構成の説明に紙面を割いており、投資対効果やビジネス上の意義が書かれていない

#### 7. [情報源の欠如] SFAツールの導入費用に情報源がない
- **該当箇所**: 「Salesforceの導入費用は年間200万円、初期構築に150万円を見込む」
- **問題**: 具体的な金額が記載されているが、出典（公式料金ページ、見積もり結果等）が示されていない

### 推奨アウトライン

この資料は複数カテゴリにまたがる指摘があり、箇所単位の修正では全体の質が上がらないため、以下の構成で書き直すことを推奨する。

1. **提案の要旨**
   - この資料で何の承認を求めるのかを1〜2文で明記する（指摘事項のメタ情報の問題を解消）
   - 例: 「営業部門の商談管理・レポート作成を自動化するため、SalesforceのEnterprise版導入を提案する。初年度投資{金額}万円、年間削減効果{金額}万円を見込む」

2. **現状の課題（ファクトベース）**
   - 営業部門の現状を、データで裏付けて記述する（指摘3・5を解消）
   - 書くべき内容: 営業担当者の業務時間内訳（実測データ）、現在の商談管理方法とその問題点（具体的なエピソードや数値）、競合他社の動向（公開情報をソース付きで）

3. **提案内容（具体的なアクション）**
   - 「DX」のような曖昧な言葉を使わず、実施する施策を個別に定義する（指摘1・4を解消）
   - 各施策に Who / What / When / How を明記する（指摘2を解消）
   - 書くべき内容: 施策の一覧と各施策の具体的な内容、担当チーム・体制、スケジュール（マイルストーン付き）

4. **投資対効果（経営層向け）**
   - 技術詳細ではなく、ビジネスインパクトを中心に記述する（指摘6を解消）
   - 書くべき内容: 導入コスト（情報源付き）、期待される削減効果（時間・金額）、ROI試算

5. **リスクと対策**
   - 導入時のリスクとその対処方針を記述する
   - 書くべき内容: 移行期間中の業務への影響、定着しなかった場合の撤退基準
```
