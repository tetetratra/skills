---

## 目的

- タスクの内容を実装可能な作業単位（ステップ）に分解する
- 実装の順序と粒度を明確にし、手戻りを最小化する
- 各ステップのテストとコミットのタイミングを計画する

注意：このコマンドは実装そのものを行わず、「どの順番で何を作るか」を計画するものです。
実際のコード作成は次のステップで行います。
また、この計画はあくまで実装（コードとテストの作成）に関するものであり、動作確認（検証計画・検証実行）は含めません。
動作確認は実装完了後の別ステップで行います。

## 実装計画の作成手順

以下のTDD的なアプローチに従って実装計画を作成してください。

- 1. ブランチ作成
  - ブランチが切られていない場合、最初にブランチを切るステップを追加してください
  - ブランチ名は、READMEや他のスキル等に命名ルールがあればそれに従ってください
  - 無い場合、`{prefix}/{チケット番号}-{短い説明}`としてください
    - prefix には「feature」「bugfix」「hotfix」など適切なものを入れます
    - 例: `feature/123-add-user-authentication`

- 2. 骨組み実装
  - 今回の開発で追加・修正するアプリケーションコードについて、そのファイル自体やインターフェースのみを作成するステップです
  - クラス定義とpublicメソッドのシグネチャのみを実装し、中身は空やNotImplementedError等としておきます
  - コミットメッセージを考えてください
    - 例：「◯◯機能を✕✕するためのコードの骨組みを実装」

- 3. ファイルごとの実装計画
  - 今回の開発で追加・修正するアプリケーションコードそれぞれについて、以下の手順でそれぞれステップを追加してください
  - 手順
    - 3-1. テスト作成
      - 追加・修正するアプリケーションコードの単体テストを追加するステップを追加してください
      - コミットメッセージを考えてください
        - 例：「(クラス名など)のテストを追加」
    - 3-2. 実装
      - 手順2で追加した枠組みの、中身の実装を埋めるステップを追加してください
      - コミットメッセージを考えてください
        - 例：「(クラス名など)の処理を実装」

- 4. その他のファイルの実装計画
  - 手順3で扱わなかったファイルについて、実装のステップを追加してください

- 5. 統合テストの作成
  - 必要に応じて、統合テストを追加・修正するステップを追加してください

- 6. リファクタリングのステップを追加
  - 最後に「リファクタリング」というステップを追加してください
  - 具体的なリファクタリング内容は実装計画の段階で考える必要はありません。実際に実装に着手する際に検討してください

- 7. lintやフォーマットの修正を行うステップを追加


### 指針

- コミットメッセージの書き方は、コミットメッセージのガイドに関するスキルを参考にしてください

## 出力フォーマット

作成した実装計画をは、以下のフォーマットで出力してください。

- ファイルパスには `追加:` （新規ファイル）または `修正:` （既存ファイル）等のプレフィックスを付ける
- テストのステップでは、テストの階層構造を示す
- 実装のステップでは、メソッド名（publicメソッドのみ）とその処理内容を木構造風に示す
- コミットを伴うステップには、コミットメッセージ案を含める
  - ブランチ作成や、実施有無を判断するだけのステップなど、コミットを伴わないステップには不要
  - 詳細なコミットメッセージが必要な場合は `commit-message-guide` を参照する
  - このスキルの出力では、1行サマリのコミットメッセージ案でもよい

```
## 実装計画

- {ステップ名}
  - 追加/修正: `{ファイルパス}`
    - {そのファイルのどの関数にどのような改修を加えるかを自然言語で具体的に書く}
  - コミットメッセージ: {コミットを伴う場合のみ書く}
- ...
```

## 出力例

管理画面からユーザーへの一括通知機能を追加する場合の例:

```
## 実装計画

- ブランチ作成
  - ブランチ名: `feature/123-add-bulk-user-notification`

- 一括ユーザー通知機能の骨組み作成
  - 追加: `app/services/bulk_user_notifier.rb`
    - クラス定義と `#call` メソッドのシグネチャのみを追加する
  - 修正: `app/controllers/admin/users_controller.rb`
    - `#notify_all` アクションのシグネチャのみ追加
  - コミットメッセージ: `一括ユーザー通知機能の骨組みを作成`

- BulkUserNotifierのテスト作成
  - 追加: `spec/services/bulk_user_notifier_spec.rb`
    - テスト構造:
      - describe "#call"
        - context "対象ユーザーが複数件の場合"
          - it "全員に通知が送信されること"
        - context "対象ユーザーが0件の場合"
          - it "エラーなく正常完了すること"
  - コミットメッセージ: `BulkUserNotifierのテストを追加`

- BulkUserNotifierの実装
  - 修正: `app/services/bulk_user_notifier.rb`
    - メソッド構成:
      - `#call`
        - 対象ユーザーを取得する
        - 通知対象ユーザーの状態を更新する
        - 対象ユーザーへ通知を送信する
  - コミットメッセージ: `BulkUserNotifierの処理を実装`

- notify_allアクションのテスト作成
  - 修正: `spec/controllers/admin/users_controller_spec.rb`
    - テスト構造:
      - describe "POST #notify_all"
        - context "正常系"
          - it "BulkUserNotifierを呼び出しリダイレクトすること"
        - context "権限がない場合"
          - it "403を返すこと"
  - コミットメッセージ: `notify_allアクションのテストを追加`

- notify_allアクションの実装
  - 修正: `app/controllers/admin/users_controller.rb`
    - `#notify_all`
      - リクエストを受けて一括通知対象の条件を組み立てる
      - BulkUserNotifier を呼び出して通知処理を実行する
      - 実行結果に応じて完了メッセージを設定し、一覧画面へリダイレクトする
  - コミットメッセージ: `notify_allアクションの処理を実装`

- ルーティングの追加
  - 修正: `config/routes.rb`
    - admin/users に `post :notify_all` を追加する
  - コミットメッセージ: `一括通知用のルーティングを追加`

- 統合テストの作成
  - 修正: `spec/requests/admin/users_spec.rb`
    - テスト構造:
      - describe "POST /admin/users/notify_all"
        - context "管理者が一括通知を実行する場合"
          - it "一括通知が実行され完了画面へ遷移すること"
  - コミットメッセージ: `一括通知機能の統合テストを追加`

- リファクタリング
  - 実装に着手した段階で、実施が必要かどうかを判断する
  - 実施する場合は、その時点で対象ファイルと具体的な整理内容を決める
  - コミットメッセージ: `一括通知機能をリファクタリング`

- lintやフォーマットの修正
  - 実装に着手した段階で、lintやフォーマッタの実行有無を判断する
  - 実施する場合は、検出された指摘内容に応じて対象ファイルと修正内容を決める
  - コミットメッセージ: `コードのlintとフォーマットを修正`
```
