## 目的

- タスクを、安全に実装できる作業単位へ分解する
- 実装順序、テスト方針、コミット粒度を明確にする
- レビューしやすく、各コミットで壊れない実装計画を作る

このスキルでは実装そのものは行わず、「どの順番で何を作るか」を計画します。
動作確認の詳細な検証計画・検証実行は、実装後の別工程で扱います。

## 作成手順

実装計画は、変更内容に合う安全なコミット単位で作成してください。

### 1. 変更タイプを判定する

まず、今回の変更が主にどれに該当するかを判断してください。

- 新規機能
- 既存機能の変更
- バグ修正
- リファクタリング
- 設定・ドキュメント・CI変更

複数にまたがる場合は、主要な変更タイプを決めたうえで、必要に応じて作業単位内で補足してください。

### 2. タスクを作業単位へ分解する

変更タイプに応じて、タスクを実装しやすい作業単位へ分解してください。

- 新規機能: 最小の骨組み → 振る舞い単位の実装/テスト → 接続 → 整理
- 既存機能の変更: 既存挙動の確認 → 必要な保護テスト → 実装 → 影響範囲確認
- バグ修正: 再現テスト → 修正 → 回帰確認
- リファクタリング: 既存テスト確認 → 小さな構造変更 → 挙動差分なし確認
- 設定・ドキュメント・CI変更: 現状確認 → 変更 → 実行/表示確認

### 3. 各作業単位の内容を具体化する

各作業単位には、次の内容を含めてください。

- 追加/修正/削除するファイル
  - 対象ファイルと具体的な変更内容を書く
- テスト/確認方法
  - 実行するコマンドや確認内容を書く（不要な場合はその旨を書く）
  - ここで書く確認は作業単位ごとの確認であり、実装後の総合的な動作確認とは別に扱う
  - 再現テストなど、失敗すること自体を確認する場合は期待する失敗内容を書く
- コミット
  - この作業単位をコミットするかを判断し、する場合はコミットメッセージ案も書く

ブランチ作成や実施有無の判断だけの作業単位など、コミットを伴わない場合はコミットメッセージ案は不要です。
コミットメッセージの書き方は、必要に応じて `commit-message-guide` を参考にしてください。

## 判断基準

- 固定のTDD手順ではなく、変更内容に合う順序を選んでください
- ファイル単位ではなく、責務・振る舞い・リスク・レビューしやすさを基準に分解してください
- 形式的な骨組み実装、形式的なテスト分離、形式的なリファクタリング作業は入れないでください
- コード変更とテスト追加を分けるか同じ作業単位にするかは、コミットの目的と作業後の確認内容を基準に判断してください
- 原則として、コミット時点で関連するテスト/ビルドが通る状態にしてください
- ただし、再現テストなど失敗状態を残す目的がある場合は、テスト/確認方法に期待する失敗内容を明記してください
- 最終的な実装完了時点では、関連するテスト/ビルドが通る計画にしてください

## 出力前チェック

計画を出力する前に、以下を確認してください。

- 変更タイプに合う分解になっているか
- 作業単位の順序が依存関係に沿っているか
- 各作業単位がレビュー可能な意味のある単位になっているか
- テスト/確認方法が空欄や曖昧な表現になっていないか
- 不要な骨組み実装、形式的なテスト分離、形式的なリファクタリング作業が入っていないか

## 出力フォーマット

```
## 実装計画

変更タイプ: {新規機能 / 既存機能の変更 / バグ修正 / リファクタリング / 設定・ドキュメント・CI変更}

分解方針: {この順序で進める理由を1〜2文で書く}

- {作業単位名}
  - 追加/修正/削除: `{ファイルパス}`
    - {具体的な変更内容}
  - テスト/確認方法: `{実行するコマンドや確認内容 / なし}`
  - コミット: {コミットメッセージ / なし}
```

必要に応じて、1つの作業単位に複数ファイルを含めて構いません。
ただし、その場合も作業単位全体が1つの意味のある変更単位になっている必要があります。
