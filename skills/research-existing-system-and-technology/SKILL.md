## 目的

- 今回のタスクに関連する既存システムの仕組みを把握する
- 今回のタスクに関連する技術・概念・前例を調査する
- 後続の要求定義、要件定義、対応方針決めに必要な判断材料を揃える

このスキルでは調査結果の整理までを行います。
対応方針の決定や詳細設計は行わないでください。

## 手順

1. まず、以下の候補をもとに、今回の調査をどのまとめる際の視点をいくつか選んでください
  - 候補
    - 既存システムの概要（原則として含めてほしいです）
    - 動作フロー
    - データ構造・テーブル構成
    - 関連技術
    - 類似実装・前例
  - 小見出しはユーザーの負担を考慮して最低限にしてください
    - 基本的には1個以上、数個以内に収めてください
2. 各視点ごとに、必要なコード、既存ドキュメント、チケット、参考資料を調査してください
  - `関連技術` は、アプリケーションの仕様説明ではなく、今回の理解に必要な一般的な技術や概念を扱ってください
  - その技術が広く知られた一般的なものなら無理に書く必要はありません
  - `類似実装・前例` も、今回の理解に役立つものがある場合にだけ選んでください
3. 各視点ごとに、調査結果を小見出し内へ整理してください
  - 事実・推測・未確認事項が混ざらないようにしてください
  - 推測が入る場合は、その根拠と推測であることを明記してください
  - `事実:` や `根拠:` のような不自然な接頭辞は付けず、自然な文章で整理してください
  - 調査メモの断片を並べるのではなく、短い前提整理ドキュメントのつもりで、少し丁寧に・少し構造化して書いてください
  - 1つの箇条書きでは、最初に全体像や結論を1文で述べ、その後に必要な補足を続けてください
  - 細かな実装識別子をそのまま並べるのではなく、まずはユーザーが追える単位で説明してください
  - 関数名や変数名は、ユーザーが把握していないことが多いため、基本的には書かないでください
  - 画面名、機能名、コンポーネント名、モデル名など、ユーザーが把握している可能性が高い名前は、必要な範囲で書いて構いません
  - 図示したほうが分かりやすい箇所は、mermaid などで図示してください
4. 調査の途中で新しい論点が見つかった場合は、必要に応じて視点を追加・修正してください

## 出力フォーマット

```
## 調査結果

### {小見出し}
- {調査結果}
  - {必要に応じて補足}
  - ...
- ...
```

## 出力例

コードブロック内にコードブロックを置くと表示が崩れるので、例ではバッククォート2個で表現しています。

```
## 調査結果

### 既存システムの概要
- 質問投稿機能は、投稿画面で受け付けた内容を `questions` テーブルへ保存し、モデレーター画面の確認後に一覧へ反映する構成
  - 投稿内容は `Question` モデルとして扱われ、投稿直後に全体公開するのではなく、`status` カラムが `published` になっているレコードだけを表示対象にする前提
- ...

### 動作フロー
- 視聴者が投稿画面から質問を送信すると、バックエンドで `Question` レコードが作成され、モデレーター画面から確認できる状態になる
  - 保存時点では公開状態（`status: published`）にならず、確認待ち（`status: pending`）の扱い
- モデレーターが承認（`Admin::QuestionsController#approve`）すると、公開対象の一覧に含まれる
  - 承認操作を起点に、質問一覧コンポーネントから見える公開対象へ反映
- ...
``
flowchart LR
  A[質問投稿画面] --> B[保存<br/>Question レコード作成]
  B --> C[モデレーター画面で確認]
  C --> D[承認<br/>Admin::QuestionsController#approve]
  D --> E[公開一覧に反映<br/>status: published]
``
```

## 注意点

- 対応方針決めなどはこのスキルでは行いません。出力しないでください
- 調査結果と意思決定を混ぜないでください
- 事実に基づく説明を優先し、判断が必要な点は未解決事項として補足的に明記してください
