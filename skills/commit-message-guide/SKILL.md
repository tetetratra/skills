## コミットメッセージの指針

以下の構造で、初見の人が追加の質問なく理解できるコミットメッセージを書く。

### 構造

1行目: [プレフィックス] 1行サマリ（何を変えたか）

（空行）

このコードが何のためのものか、どこから呼ばれるかを1〜2文で説明する。

なぜ変更が必要かを説明する。エラーや問題が起きていた場合は、
どのような条件で何が起きていたかを具体的に書く。
技術的な原因（制約、仕様の違い等）も含める。

- 変更内容: 変更前 → 変更後（箇条書き）

なぜこの変更で問題ないと言えるかを1〜2文で説明する。
影響範囲の確認結果（呼び出し元の使われ方等）があれば含める。

### 指針

- **対象コードの役割を書く**: 変更箇所が何のためのコードか、どの画面・機能で使われるかを最初に説明する
- **問題の現象を書く**: 「エラーになる（具体的な内容と共に）」「順序が保証されない」等、何が起きていたかを明示する
- **技術的な原因を書く**: なぜそれが問題になるのかのメカニズムを説明する
- **変更の安全性を書く**: 呼び出し元の確認結果や、挙動が変わらない根拠を示す
- **変更前→変更後を明示する**: 箇条書きで差分を簡潔に示す
- 本文は日本語で書く
- 1行は80文字程度で折り返す

### 文言例

バグ修正

```
[修正] ユーザー一覧の検索結果が重複して表示される問題を修正

ユーザー一覧画面の検索機能は、名前・メールアドレスの部分一致で
フィルタリングするもので、管理画面から使用される。

複数のロールを持つユーザーを検索すると、rolesテーブルとの
JOINにより同一ユーザーが重複して返されていた。

- User.joins(:roles).where(...) → User.joins(:roles).where(...).distinct

rolesはフィルタ条件としてのみ使用しており、集計（count等）は
別のクエリで行っているため、DISTINCTの追加で他の処理に影響はない。
```

リファクタリング

```
N+1クエリの解消: 注文一覧での商品情報取得をeager loadに変更

注文一覧画面では、各注文に紐づく商品名と価格を表示しており、
管理者が日次の注文確認で使用する。

注文が100件ある場合、商品情報の取得で100回のSQLクエリが
発行されており、ページ表示に3秒以上かかっていた。

- Order.all → Order.includes(:items)
- ビューでのorder.items.eachは変更なし（eager loadが適用される）

includesにより1+1の2クエリに削減される。ビュー側のコードは
変更不要で、表示内容も変わらない。
```

機能追加

```
メール通知にCSVエクスポート完了通知を追加

大量データのCSVエクスポートは非同期ジョブで実行されるが、
完了を知る手段がなく、ユーザーが画面を繰り返しリロードしていた。

- ExportJob完了時にExportMailer.completed通知を送信するよう追加
```
