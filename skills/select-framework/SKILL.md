---
description: 直面している問題に対して、適用すべき思考フレームワークを1〜3個提案する
---

## 目的

ユーザーが直面している問題(意思決定・分析・改善・伝達・計画・目標設定・役割分担・モチベーション など)に対して、適用すべき思考フレームワークを1〜3個提案します。

このskillはフレームワークの **選定** までを担います。提案時に簡易な適用ステップは示しますが、深い適用支援(実際にフレームワークを使った詳細分析を伴走するなど)は範囲外です。

## 入力

skill起動時の引数として、ユーザーから問題情報(自然文)を受け取ります。
空入力の場合は、まず「どんな問題に直面しているか」を質問してから進めてください。

## 進め方

以下のフローチャートに従ってください。

### Step 1: 問題情報の受け取り

- 入力が空のとき → ユーザーに「どんな問題で悩んでいるか」を質問し、回答を得てから Step 2 へ進みます
- 入力があるとき → Step 2 へ進みます

### Step 2: 「種類」「対象」の判別

問題情報から、以下の2軸を判別してください。

**種類**(=何をしたいか):
- 意思決定: 何かを選ぶ、優先順位を決める
- 分析・理解: 既存のものを把握する
- 改善: 既存のもののムリ・ムダ・効率を直す
- 伝達: 他者に説明・浸透させる
- 計画: 作業の順序・依存・スケジュールを立てる
- 目標設定: 目指す状態を定義・分解・具体化する
- 役割分担: 誰が何をやるかを明確にする
- モチベーション: やる気・満足度を分析する

**対象**(=誰/何に対して):
- 個人タスク: 自分1人の作業
- チーム: 複数人の協働
- 業務フロー: 反復される手順・運用
- サービス: 機能やプロダクト全体
- ステークホルダー: 上司・他部署・顧客などの利害関係者

判別結果の分岐:
- 両方判別できた → Step 4 へ進みます
- 一方または両方が不明 → Step 3 へ進みます

### Step 3: 逆質問(1往復まで)

判別に必要な情報をユーザーから引き出すため、最大1往復の質問を行ってください。
複数の質問は1メッセージにまとめます。例:

> 状況をもう少し教えてください:
> - これは個人で悩んでいることですか? それともチーム・業務全体に関わることですか?
> - 主にやりたいのは「優先順位を決める」「現状を理解する」「改善する」「人に伝える」のどれに近いですか?

回答を得たら Step 2 に戻ります。
**2回目以降は判別不能でも推測で Step 4 へ進んでください**(質問を繰り返さないでください)。

### Step 4: フレームワーク選定

「フレームワーク一覧」の判定タグと、Step 2 で判別した「種類×対象」を照合し、合致度の高いものを1〜3個選んでください。

選定の指針:
- 単独で問題を解決できるとき → 1個
- 補完関係があるとき(例: 「PERT図」で全体計画 + 「RACI」で役割分担 / 「KPIツリー」で目標分解 + 「SMART」で各指標を具体化) → 2〜3個
- 3個を超えないようにしてください(多すぎると判断に迷うため)

### Step 5: 提案出力

「出力フォーマット」に従ってフレームワークを提案してください。

## フレームワーク一覧

以下に記載されていないフレームワークであっても、対象の課題に合致していれば提案して構いません。

### 緊急度・重要度マトリクス
- **概要**: やることを「緊急か」「重要か」の2軸で整理する考え方です。目先の対応に追われすぎず、本当に時間を使うべきことを見分けるために使います。
- **活用機会**: 障害対応、技術的負債の解消、将来の保守性向上などが同時に並ぶ場面で役立ちます。日々のタスク整理や、チームで優先順位をそろえるときの土台にもなります。
- **判定タグ**: 種類=意思決定(優先順位付け) / 対象=個人タスク, チーム

### GROWモデル
- **概要**: 意思決定を「成果物定義 → 現状把握 → 判断軸の整理(優先度含む) → 選択肢の列挙 → 各案のメリット・デメリット整理 → 意思決定」の流れで進める考え方です。一般的なGROWモデル(Goal / Reality / Options / Will)を、意思決定プロセス向けに拡張した形になります。
- **活用機会**: 技術選定、設計方針の判断、データ仕様の決定など、複数の選択肢から1つを選ぶ場面で役立ちます。「なんとなく選ぶ」のではなく、判断軸を明示してから選択肢を比較したいときに有効です。
- **判定タグ**: 種類=意思決定(過程の整理) / 対象=個人タスク, チーム

### バリューチェーン分析の社内運用版
- **概要**: サービスや業務を「ひと」「もの」「金」の流れから把握する見方です。誰が何を受け渡し、どこで価値やコストが発生しているかを整理するための枠組みです。
- **活用機会**: 既存システムや既存サービスの理解、業務フローの棚卸し、改善ポイントの発見に向いています。特に、仕様だけでは見えない社内オペレーションや部署間連携を理解するときに有効です。
- **判定タグ**: 種類=分析・理解 / 対象=サービス, 業務フロー

### AIDMA
- **概要**: 人が Attention → Interest → Desire → Memory → Action の流れで動く、という考え方です。何かを浸透させるときに、いきなり行動を求めず段階的に考えるための枠組みです。
- **活用機会**: 新しい開発ルール、パフォーマンスチェック文化、社内ツール導入などをチームへ展開するときに使えます。「なぜ広まらないのか」を分解して考えやすくなります。
- **判定タグ**: 種類=伝達(浸透) / 対象=チーム, ステークホルダー

### KPIツリー
- **概要**: 最終目標を、その手前の指標へ分解して木構造で整理する方法です。目標と日々の行動指標のつながりを見える化するために使います。
- **活用機会**: 性能改善、開発生産性改善、障害削減などを進めるときに役立ちます。たとえば「リリース品質向上」を、レビュー時間・バグ件数・テスト整備率などへ落とすときに使いやすいです。
- **判定タグ**: 種類=目標設定(分解) / 対象=サービス, チーム

### SMART
- **概要**: 目標を Specific, Measurable, Achievable, Relevant, Time-bound の観点で点検する考え方です。曖昧な目標を、実行しやすく評価しやすい形に整えるためのものです。
- **活用機会**: 個人目標、チーム目標、改善施策の目標設定に向いています。「性能を良くする」ではなく、「今月中にAPI p95を20%改善する」のように具体化するときに使えます。
- **判定タグ**: 種類=目標設定(具体化) / 対象=個人タスク, チーム

### PERT図
- **概要**: 作業の順序や依存関係を図にして、どの経路が全体の完了時期を左右するかを見る手法です。計画の見通しとボトルネック把握に向いています。
- **活用機会**: 大きめの開発案件、移行作業、負荷試験基盤づくりのように、複数作業が依存し合う場面で使えます。どこが遅れると全体に効くかを見つけやすいです。
- **判定タグ**: 種類=計画 / 対象=サービス, 業務フロー

### RACI
- **概要**: 役割分担を Responsible, Accountable, Consulted, Informed に分けて整理する枠組みです。誰が作業担当で、誰が最終責任を持つかを明確にします。
- **活用機会**: 新規プロジェクト開始時、横断的な改善活動、インフラや運用を含む施策で役立ちます。責任の押しつけ合いや、関係者の認識ズレを減らしやすいです。
- **判定タグ**: 種類=役割分担 / 対象=チーム, ステークホルダー

### ムリ・ムラ・ムダ
- **概要**: 業務の問題を「無理」「ばらつき」「無駄」の観点から捉える改善の考え方です。現場のつらさや非効率を、感覚ではなく観点で整理できます。
- **活用機会**: 運用フロー、レビュー体制、障害対応、属人化した作業の見直しに向いています。特定メンバーだけに負荷が偏っている、手順が毎回違う、不要な確認が多い、といった問題の発見に使えます。
- **判定タグ**: 種類=改善(問題発見) / 対象=業務フロー

### ECRS
- **概要**: Eliminate, Combine, Rearrange, Simplify の順で改善を考える型です。まず消せないかを考え、それがだめならまとめる・並べ替える・単純化する、という順に検討します。
- **活用機会**: 定例作業、リリース手順、レビュー依頼フロー、監視対応手順などの改善に役立ちます。単に頑張るのではなく、作業そのものを減らす発想を持ち込みやすいです。
- **判定タグ**: 種類=改善(作業削減) / 対象=業務フロー

### 動機づけ・衛生理論
- **概要**: 人の満足を高める要因と、不満を減らす要因は別だと考える理論です。待遇や環境だけでなく、達成感や成長実感も重要だと捉えます。
- **活用機会**: 自分の働き方やモチベーションの分析、チームの状態理解に使えます。「不満は少ないのにやる気も出ない」といった状態を整理するときに役立ちます。
- **判定タグ**: 種類=モチベーション分析 / 対象=個人タスク, チーム

### PREP
- **概要**: Point → Reason → Example → Point の順で伝える、説明の型です。結論先行で、短くても伝わりやすい文章や会話を作りやすいです。
- **活用機会**: 技術提案、改善提案、レビューコメント、上司や他部署への説明などで使いやすいです。特にビジネス寄りの場面で、「何を言いたいのか」を先に伝えたいときに有効です。
- **判定タグ**: 種類=伝達(説明の型) / 対象=ステークホルダー

## 出力フォーマット

```
## 提案フレームワーク

(複数提案する場合は、ここに補完関係を一行で記載してください。例: 「PERT図で全体計画を作り、RACIで各作業の役割を明確化することで補完的に使えます」)

### 1. {フレームワーク名}
- **概要**: {1-2行}
- **なぜ合うか**: {今回の問題への適合理由を1-3行}
- **適用ステップ**:
  1. {今回の問題に当てはめた具体的ステップ}
  2. ...
  3. ... (合計3〜5ステップ)

### 2. {フレームワーク名}
(同上、複数提案の場合のみ)
```

## チェックポイント

- [ ] 提案するフレームワークは1〜3個に収まっていますか
- [ ] 逆質問は1往復までに収まっていますか
- [ ] 「適用ステップ」は今回の問題に当てはめた具体的内容になっていますか(フレームワーク自体の一般説明に終始していませんか)
- [ ] 複数提案する場合、フレームワーク間の補完関係を冒頭で示していますか
- [ ] 各提案で「なぜ合うか」を、問題の文脈に即して述べていますか
