デモ・モックアプリをAIと協力して0から作成する際の推奨ワークフローです。

このスキルはAIが参照し、ユーザーと対話しながら各ステップを順に進めるためのガイドです。各ステップでは計画と実装を分離し、ユーザーの承認を得てから次に進んでください。

**重要な原則**:
- 各ステップの計画がユーザーに承認されるまで、コードの変更を行わないこと。
- 1つの「やること」につき1つのチケットを作成し、進捗を管理すること。

全体は「準備フェーズ（1回ずつ）」「基盤構築（1回）」「機能開発サイクル（機能ごとに繰り返し）」で構成されます。各ステップの深さはスコープに応じて調整してください。ユーザーが既に明確なイメージを持っている場合、リサーチは軽くて構いません。

## 準備フェーズ

### リサーチ

**目的**: アプリの方向性を把握し、設計判断に必要な前提知識を揃える。

**やること**

- 意図の聞き取り
  - ユーザーから「何を作りたいか」のざっくりした方向性を把握する
- 類似サービスの調査
  - 参考になるアプリ・サービスがあれば調査し、機能・UIの傾向を把握する
- ドメイン知識の整理
  - 対象ドメインの前提知識（用語、業務フロー等）を整理する
- 不明点の洗い出し
  - 曖昧な点をリストアップし、ユーザーに質問する

### 目的・コア機能の定義

**目的**: デモで見せたい価値と範囲を確定し、以降の設計・実装の判断基準を作る。

**やること**

- 目的の整理
  - アプリの目的（誰の・何の課題を・どう解決するか）を定める
- コア機能の一覧化
  - 各機能を一言で説明する。網羅性より「触って価値が伝わる」動線を優先する
- 非目標の明示
  - このデモでは扱わないことを明示的に除外する
- 受け入れ基準の定義
  - 各コア機能について「何ができれば完成か」をチェックボックス形式で記述する

**参考にするスキル**

- ビジネス的背景・目的を理解するスキル

### 技術スタック・全体設計

**目的**: 技術的な前提とルールを確定し、AIが一貫した判断で実装できる状態を作る。

**やること**

- 技術スタックの選定
  - FE/BE/DB/認証/デプロイ先を決め、選定理由を記録する
- ディレクトリ構成
  - プロジェクトのディレクトリ構成を定義する

**参考にするスキル**

- 方針を決めるスキル

### 環境構築

**目的**: 実装を開始できる空のプロジェクトを用意し、ベースラインを確保する。

**やること**

- プロジェクト初期化
  - `npm init` / `create-next-app` 等でプロジェクトを作成する
- 依存パッケージのインストール
  - 「技術スタック・全体設計」で選定したパッケージを導入する
- ディレクトリのスキャフォールディング
  - 「技術スタック・全体設計」で定義した構成に沿ってディレクトリを作成する
- 開発ツールのセットアップ
  - リンター・フォーマッター・テストフレームワークを設定する
- 初回コミット
  - ビルド・テスト・リントが通る空の状態を確保してコミットする

**参考にするスキル**

- 実装計画を立てるスキル

## 基盤構築

準備フェーズ完了後、機能開発サイクルに入る前に、共通基盤を構築する。ここでは具体的な機能の実装は行わず、機能を載せるための土台づくりに徹する。

### 基盤の定義

**目的**: 機能開発で共通して必要となる土台を洗い出し、構築範囲を確定する。

**やること**

- 共通UI要素の洗い出し
  - レイアウト（ヘッダー、サイドバー、フッター等）、ナビゲーション、TOPページなど
- 技術的な共通基盤の洗い出し
  - ルーティング設定、APIクライアント共通設定、エラーハンドリング等
- 構築範囲の確定
  - 機能開発で必要になる最小限の範囲に絞り、過剰な先行投資を避ける

### 基盤の実装

**目的**: 確定した共通基盤を実装し、機能開発をすぐ始められる状態にする。

**やること**

- 「基盤の定義」で確定した範囲を実装する
- 各機能のページが配置できるルーティングとレイアウトが動作する状態をゴールとする
- コミット

**参考にするスキル**

- 設計を立てるスキル
- 実装計画を立てるスキル

## 機能開発サイクル

以下を、機能ごとにVertical Slice（DB→API→UIを一気通貫）で繰り返す。機能間に依存関係がある場合は、依存される側から順に着手すること。

### 機能の詳細定義

**目的**: 対象機能の振る舞いを具体化し、実装とテストの判断基準を作る。

**やること**

- ユーザーストーリー
  - 「〜として、〜したい、〜のために」形式で記述する
- 画面構成・画面遷移
  - 画面の構成要素と画面間の遷移を定義する
- 入力・バリデーション
  - 入力項目とバリデーションルールを定める
- 振る舞いの定義
  - 正常系・異常系の振る舞いを記述する
- エッジケース
  - 想定されるエッジケースを洗い出す

**参考にするスキル**

- 要件定義を行うスキル

### UIモックの作成

UIを伴う機能の場合に実施する。バックエンド処理のみの機能ではスキップしてよい。

**目的**: バックエンド実装前に見た目と操作感の合意を取り、手戻りを防ぐ。

**やること**

- 「機能の詳細定義」をもとに、静的なUIモック（HTML/コンポーネント）を先に作成する
- ユーザーに見た目と操作感を確認してもらい、合意を得てから技術設計に進む

### 機能の技術設計 + 実装計画

**目的**: 実装の進め方を具体化し、AIが迷わず作業できる手順に落とし込む。

**やること**

- データモデル設計
  - テーブル定義、リレーションを設計する
- API設計
  - エンドポイント、リクエスト/レスポンス形式を定める
- UIコンポーネント設計
  - UIモックをもとに、コンポーネントの分割方針を決める
- 実装ステップの分解
  - 各ステップで作成・変更するファイル一覧を整理する
- 実装順序の決定
  - 各ステップの依存関係と実装順序を定める

**参考にするスキル**

- （なにか設計上の判断が求められる場合）方針を決めるスキル
- 設計を行うスキル

### 実装

**目的**: 計画に沿ってコードを書き、動作する機能を完成させる。

**やること**

- Vertical Sliceで実装
  - 「機能の技術設計 + 実装計画」に沿って、DB → API → UI の順で実装する
- ステップごとのコミット
  - 1ステップ完了ごとにコミットする
- 動作確認
  - ステップ間でテスト・型チェックを実行し、壊れていないことを確認する
- 計画との整合性維持
  - 計画との乖離が生じた場合は、計画を更新してから続行する

**参考にするスキル**

- 実装計画を行うスキル
