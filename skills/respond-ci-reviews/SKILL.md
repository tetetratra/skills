---
description: PR上のCIレビューbotのコメントに対して、自律的に回答・修正対応を行う。
---

PR上のCIレビューbot（claude、devin等）からのレビューコメント・質問に対して、
調査・回答・コード修正を自律的に行い、CI全成功まで繰り返します。

## 手順

### 1. CIレビュー完了の待機

- `gh pr checks {PR番号}` でCI状態を確認する
- レビュー系CIジョブが未完了（進行中）の場合、60秒間隔でポーリングして完了を待つ
- テスト系ジョブ（rspec、rspec_with_assets等）もIN_PROGRESSの場合は同様に待機する

### 2. レビューコメントの収集

- `gh api repos/{owner}/{repo}/pulls/{PR番号}/comments` でインラインレビューコメントを取得する
- `gh api repos/{owner}/{repo}/issues/{PR番号}/comments` でPR全体のコメントも確認する

### 3. 各コメントの分析と対応

各コメントについて以下を判断し、対応する:

#### A. コード修正が必要な場合

1. 指摘内容が正当か確認する（コードを読み、呼び出し箇所・テストを調査）
2. 修正を実装する
3. `bundle exec rubocop` 等、適切なコマンドで静的解析を通す
4. 修正をコミットにする
5. pushする
6. コメントに返信する
   - 修正内容と修正理由を簡潔に説明する

#### B. 回答のみで対応可能な場合（注意喚起・確認系）

1. 指摘内容を調査し、現時点で問題があるかを確認する
2. 調査結果を踏まえてコメントに返信する
   - 問題がない場合はその理由を説明する
   - 今後のリスクがある場合はその旨を記載する

#### C. 既に解決済みの場合

- bot自身がResolvedと判断している場合や、他のコメントで対応済みの場合はスキップする

### 4. CI結果の確認

- コード修正を行った場合、push後にCIが再実行される
- `gh pr checks {PR番号}` でCI状態を確認する
- テスト系ジョブ（rspec等）が実行中の場合、完了まで待機する（60〜300秒間隔でポーリング）
- CIが失敗した場合:
  - MCP等でCircleCI等のCIステータスを取得できる場合、失敗ログを取得する
  - 失敗原因を分析し、修正を実装する
  - 該当コミットに統合してforce pushする
  - 手順4に戻る

### 5. 完了判定

以下の条件がすべて満たされた場合に完了とする:
- 全レビューコメントに対応済み（返信済みまたはスキップ済み）
- CI全ジョブが成功（pass）
- botからの追加コメント・質問がない

条件を満たさない場合、手順1に戻ってループを継続する。

## コメント返信のガイドライン

- 簡潔で技術的な返信を心がける
- 修正した場合: 何をどう修正したかと、その理由を説明する
- 修正しない場合: なぜ現状で問題ないかを、根拠とともに説明する
- 潜在リスクの指摘に対しては: 現時点の影響と今後の注意点を説明する

## 注意事項

- `$()` 等のインラインコマンド評価を含むbashコマンドは極力使用しない（ユーザー承認が必要になるため）
- tmpファイルの無意味な作成を避ける
- コミットメッセージはコマンド引数に直接渡す
- `git rebase -i` は使用しない（インタラクティブモード非対応のため）
- コード修正前に必ず該当ファイルを読み、影響範囲を確認する

- [ ] コード修正がある場合、rubocop等のlintを通したか
- [ ] CI全ジョブが成功しているか
